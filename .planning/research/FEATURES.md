# Feature Landscape

**Domain:** Personal developer website with CMS and blog
**Researched:** 2026-03-20

---

## Table Stakes

Features users (recruiters, hiring managers, collaborators, readers) expect. Missing = site feels incomplete or unprofessional.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| About / bio section | First thing every visitor looks for — who are you, what do you do | Low | Already exists; CMS-manage it |
| Portfolio / project showcase | The primary signal of capability; must demonstrate you can build things that work | Medium | Already exists; needs richer project write-ups |
| Skills display | Hiring managers scan for tech stack fit; expected at a glance | Low | Already exists; CMS-manage it |
| Resume / CV | Standard expectation; PDF download and/or on-page content | Low | Already exists; CMS-manage it |
| Responsive / mobile-first design | 57-59% of web traffic is mobile; non-responsive = instant credibility loss | Medium | Already exists; verify in new build |
| Readable typography, sufficient whitespace | Developer aesthetic expectation; cluttered = amateur signal | Low | Design/CSS decision, not a feature |
| Dark/light mode toggle | Now an expectation on developer sites, especially for technical audiences | Medium | New requirement; system preference detection + manual toggle |
| Fast load times (LCP < 2.5s) | Google Core Web Vitals are ranking factors; slow = SEO penalty + user abandonment | Low-Med | Framework choice and image handling drive this |
| Accessible color contrast | WCAG compliance is legally required in EU (June 2025) and expected globally; contrast failures are the #1 accessibility violation | Low | Design system decision; enforce in both modes |
| Open Graph meta tags | Controls how links appear on LinkedIn, Twitter/X, Slack; missing = ugly unfurls that look unprofessional | Low | Template-level, minimal effort |
| Page titles and meta descriptions | Basic SEO; missing hurts discoverability | Low | Template-level |
| Social / contact links | Expected in footer/header; LinkedIn, GitHub at minimum | Low | CMS-managed |

## Differentiators

Features that set the site apart. Not universally expected, but signal quality and thoughtfulness.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Blog with developer-focused writing | Signals thought leadership; shows how you think, not just what you've built; builds long-term SEO equity | Medium | New requirement; key differentiator from pure portfolio sites |
| Rich project case studies | Go beyond "I used React" — problem, technical decisions, trade-offs, outcomes; 78% of decision-makers trust case studies with specifics over general claims | Medium | Enhancement to existing portfolio section; CMS-driven |
| Syntax highlighting in blog posts | Table stakes for a developer writing about code; expected by technical readers, embarrassing to omit | Low | Library-level decision (Shiki or Prism); integrates at CMS/rendering layer |
| Estimated reading time on blog posts | Small quality signal that readers appreciate; signals professionalism | Low | Simple word-count computation |
| RSS / Atom feed | Signals blog permanence and respects power readers; developer audiences still use feed readers; ~30% of tech blogs offer feeds | Low | Auto-generated; worthwhile for audience signal |
| Project technology tags / filtering | Lets visitors quickly filter by tech they care about (e.g., "show me React projects") | Low-Med | Useful at 6+ projects; add when portfolio grows |
| Content-first information hierarchy | Taste and restraint as demonstrated skill; a fast, clean, scannable site is itself a portfolio piece | Low | Design decision, not a feature — but explicitly worth noting |
| Sitemap.xml | Helps Google index all pages, especially blog posts; minor effort, meaningful SEO benefit | Low | Auto-generated by most frameworks |

## Anti-Features

Features to deliberately NOT build. These waste time, add complexity, or actively harm the site.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| Comments system | Moderation overhead, spam risk, privacy implications; explicitly out of scope in PROJECT.md; adds no meaningful value for a solo dev site in v1 | Link to replies on Twitter/X or LinkedIn, or a GitHub Discussions thread if wanted |
| Custom animations / transitions / 3D effects | Looks impressive to other developers building portfolios, not to hiring managers; can signal over-engineering; adds performance cost; explicitly out of scope per PROJECT.md | Clean CSS transitions (hover states, focus rings) only |
| User accounts / authentication | No use case on a public-facing content site; significant security and maintenance burden | Public-only; CMS handles auth internally |
| Newsletter / email capture | Premature — blog needs to exist and have an audience before this is worth the tooling complexity | Add as v2 if blog gains traction |
| Contact form | Spam magnet without CAPTCHA; CAPTCHA adds friction; email is simpler and equally effective for a solo dev | Prominent email link + social links; mailto: is sufficient |
| Search (site-wide) | Low value at small content volume; adds meaningful complexity (Algolia, Fuse.js, etc.); not worth it until 20+ blog posts | Add in a later phase when blog content justifies it |
| Testimonials / endorsements section | Adds social proof but requires maintaining; rarely convincing without context; adds page clutter | Let project case studies speak for quality |
| Multiple language / i18n support | No indication of need; significant complexity multiplier for all content types | English only; revisit if there is a clear reason |
| Visitor counters / social share buttons | Visual noise; share buttons add third-party scripts and privacy implications; Open Graph tags handle sharing without the buttons | OG tags only; no share buttons |
| Portfolio without project write-ups | A list of project names with no context answers nothing; hiring managers skip it | Every project needs at minimum: what it is, the problem it solved, key tech decisions |

## Feature Dependencies

```
CMS (core) → About/bio editing
CMS (core) → Portfolio project editing
CMS (core) → Skills editing
CMS (core) → Resume/experience editing
CMS (core) → Blog post creation

Blog → Syntax highlighting (blog must exist first)
Blog → Reading time (blog must exist first)
Blog → RSS feed (blog must exist first)
Blog → Open Graph per-post (extends global OG)

Dark mode → Color contrast validation (must verify both modes)
Dark mode → Syntax highlighting theme (needs a dark theme variant)

Portfolio projects → Project case studies (enhancement to existing feature)
Portfolio projects → Technology tag filtering (only valuable at 6+ projects)
```

---

## MVP Recommendation

Build in this order, as each group depends on the previous:

**Phase 1 — Foundation + Feature Parity (no code changes to update content)**
1. About/bio (CMS-managed)
2. Skills display (CMS-managed)
3. Resume/CV (CMS-managed)
4. Portfolio project showcases (CMS-managed)
5. Responsive design + mobile nav
6. Dark/light mode toggle
7. Analytics

**Phase 2 — Blog (the primary new capability)**
1. Blog post list page
2. Blog post detail page with syntax highlighting and reading time
3. RSS feed
4. Per-post Open Graph meta

**Phase 3 — Polish and SEO**
1. Sitemap.xml
2. Meta descriptions per page/post
3. Project case study depth (richer write-ups in CMS)

**Defer:**
- Technology tag filtering: add when portfolio has 6+ projects
- Site search: add when blog has 20+ posts
- Newsletter integration: add when blog has an established readership
