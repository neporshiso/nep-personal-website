## Architecture Patterns

### Content File Format

All entries follow this established pattern:

**Blog post (`.mdoc`):**
```
src/content/posts/{slug}.mdoc
```
Frontmatter fields (from `content.config.ts`):
- `title: string` (required)
- `publishedDate: string` — ISO date `'YYYY-MM-DD'`
- `draft: boolean` — `false` for published
- `tags: string[]`
- `excerpt: string` (optional but strongly recommended)
- `coverImage: string` (optional — no cover images for these posts)

Body: Markdoc prose (headings `##`/`###`, paragraphs, fenced code blocks with language tag, bullet lists).

**Podcast entry (`.mdoc`):**
```
src/content/podcasts/{slug}.mdoc
```
Frontmatter fields:
- `name: string` (required)
- `link: string` (URL, required)
- `coverImage: string` (optional) — value must be `/assets/podcasts/{slug}.jpg`
- `category: string` (optional)

Body: empty (no body content for podcasts — matches `syntax.mdoc` and `shop-talk-show.mdoc` pattern).

**Book entry (`.mdoc`):**
```
src/content/books/{slug}.mdoc
```
Frontmatter fields:
- `title: string` (required)
- `author: string` (required)
- `status: 'reading' | 'read' | 'want-to-read'` (default `want-to-read`)
- `coverImage: string` (optional) — value must be `/assets/books/{slug}.jpg`
- `note: string` (optional)

Body: empty (no body content for books — matches `atomic-habits.mdoc` pattern).

### Asset Storage Pattern
```
public/
└── assets/
    ├── podcasts/     # .gitkeep exists; podcast covers go here
    │   └── {slug}.jpg
    ├── books/        # .gitkeep exists; book covers go here
    │   └── {slug}.jpg
    └── projects/     # .gitkeep exists (not touched this phase)
```

Images placed in `public/` are served at `/assets/...` by Astro's static file server. `coverImage: /assets/podcasts/{slug}.jpg` in frontmatter maps directly to the public path.

### Slug Derivation
Astro content collection slugs are derived from the file name (minus extension). Use kebab-case. Example:
- "Syntax - Tasty Web Development Treats" → `syntax.mdoc` → slug `syntax`
- "ShopTalk Show" → `shop-talk-show.mdoc` → slug `shop-talk-show`

Slugs for the 3 blog posts must match the file names that replace the fabricated ones. The fabricated posts have slugs:
- `building-a-personal-website-with-gatsby` → DELETE
- `learning-to-code-in-public` → DELETE
- `reflections-on-being-a-self-taught-developer` → DELETE

New real post file names/slugs:
- `data-structures-lesson.mdoc` (matches `/data-structures-lesson/` URL from source)
- `small-group-projects.mdoc` (matches `/small-group-projects/` URL from source)
- `welcome-to-the-blog.mdoc` (matches `/welcome-to-the-blog/` URL from source)

### Anti-Patterns to Avoid
- **Using `src` path in `coverImage`:** The `coverImage` value in frontmatter must be a public URL path `/assets/...` not a file system path `src/assets/...`. PodcastCard and BookCard pass this string directly to `<img src={coverImage}>`.
- **Forgetting Keystatic's separate image storage:** Keystatic config points image fields at `src/assets/images/podcasts` and `src/assets/images/books` — but this phase bypasses Keystatic and writes `.mdoc` files directly. Use `public/assets/` (not `src/assets/images/`) to match D-06/D-09 decisions and the established `coverImage` string pattern.
- **Not clearing the Astro content cache after deletions:** After deleting the 3 fabricated `.mdoc` files, delete `node_modules/.astro/data-store.json` before running `astro build` or `astro dev`. This prevents stale route generation (documented in project STATE.md).

## iTunes Search API

**Endpoint:** `https://itunes.apple.com/search?term={encoded name}&media=podcast&limit=1`

**Confirmed response structure** (tested live 2026-03-23):
```json
{
  "resultCount": 1,
  "results": [
    {
      "collectionName": "Syntax - Tasty Web Development Treats",
      "artworkUrl100": "https://is1-ssl.mzstatic.com/.../100x100bb.jpg",
      "artworkUrl600": "https://is1-ssl.mzstatic.com/.../600x600bb.jpg",
      "trackViewUrl": "https://podcasts.apple.com/us/podcast/...",
      "feedUrl": "https://feeds.megaphone.fm/...",
      ...
    }
  ]
}
```

**Key field:** `results[0].artworkUrl600` — always 600x600px JPEG.

**Download command per podcast:**
```bash
curl -sL "$(curl -s 'https://itunes.apple.com/search?term={encoded-name}&media=podcast&limit=1' | python3 -c "import sys,json; print(json.load(sys.stdin)['results'][0]['artworkUrl600'])")" \
  -o "/path/to/project/public/assets/podcasts/{slug}.jpg"
```

**Pitfall:** Search term specificity matters. "syntax" alone returns "The NO SYNTAX Podcast" as top result. Use more specific terms: "syntax wesbos tasty web" returns the correct Syntax FM podcast. Always verify `collectionName` before downloading.

**Rate limiting:** No rate limiting for single lookups. This is a one-time operation per podcast, so no throttling concerns.

**Auth:** No API key required.

## Open Library Covers API

**Endpoint:** `https://covers.openlibrary.org/b/isbn/{isbn}-{size}.jpg`

**Size options:**
- `S` — small thumbnail
- `M` — medium (recommended for books section)
- `L` — large (higher resolution)

**Confirmed behavior** (tested live 2026-03-23):
- Returns HTTP 302 redirect → archive.org JPEG URL
- `content-type: image/jpeg` on both initial and final response
- `access-control-allow-origin: *` (CORS open)
- Use `curl -L` to follow the redirect

**Download command per book:**
```bash
curl -sL "https://covers.openlibrary.org/b/isbn/{isbn}-L.jpg" \
  -o "/path/to/project/public/assets/books/{slug}.jpg"
```

**ISBN note:** ISBN-13 format preferred (e.g., `9780735224292`). Open Library also accepts ISBN-10.

**Fallback:** If Open Library has no cover for an ISBN, the endpoint returns a small 1x1 "no cover" image. Check file size after download — if under 1KB, the cover was not found; download manually from Amazon/Goodreads instead.

**No `isbn` field required** unless the plan explicitly uses it as a schema field. The ISBN is only needed during the one-time image download — it does not need to be stored in the `.mdoc` frontmatter unless the user wants CMS-editable ISBNs.

## Schema Change Decision: `isbn` Field

Per D-11, `isbn` is optional. Recommendation: **skip adding `isbn` to the schema** for this phase. Rationale:
- ISBN is only needed during image download (one-time operation)
- The `isbn` field would appear in Keystatic UI as an unvalidated text field with no runtime use
- If the user later wants to auto-refresh covers, that is a v2 enhancement
- Keeping schemas minimal reduces Keystatic form noise

If `isbn` is added, it requires changes to both `content.config.ts` (add `isbn: z.string().optional()`) and `keystatic.config.ts` (add `isbn: fields.text({ label: 'ISBN', ... })`). These are coordinated changes — both files have the comment "Schema must stay in sync."

## Common Pitfalls

### Pitfall 1: Astro Content Cache After Deletion
**What goes wrong:** After deleting fabricated `.mdoc` files, `astro dev` or `astro build` may still generate routes for the deleted slugs, or throw "missing entry" errors.
**Why it happens:** Astro v5 Content Layer caches content entries in `node_modules/.astro/data-store.json`. The cache is not invalidated on file deletion in all cases.
**How to avoid:** After deleting any `.mdoc` file, run `rm -f node_modules/.astro/data-store.json` before the next build or dev server start.
**Warning signs:** 404 on a new route, or old routes still loading after deletion.

### Pitfall 2: `coverImage` Path Convention
**What goes wrong:** Setting `coverImage: src/assets/images/podcasts/syntax.jpg` instead of `coverImage: /assets/podcasts/syntax.jpg`.
**Why it happens:** Keystatic config points its `fields.image` to `src/assets/images/podcasts` — but this phase writes `.mdoc` files directly (bypassing Keystatic). The two conventions are different.
**How to avoid:** Always use the public path (`/assets/{type}/{slug}.jpg`) — this is what `<img src={coverImage}>` in PodcastCard/BookCard expects. Public assets live in `public/` not `src/`.
**Warning signs:** Broken image in rendered page; 404 on image URL.

### Pitfall 3: iTunes Search Returns Wrong Podcast
**What goes wrong:** Downloading cover art for the wrong podcast because the search term is ambiguous.
**Why it happens:** iTunes search ranks by relevance, not exact name match. Common podcast names have many near-matches.
**How to avoid:** After fetching, always log `results[0].collectionName` and verify it matches before downloading the image.
**Warning signs:** Cover art looks wrong; `collectionName` in API response doesn't match expected podcast name.

### Pitfall 4: Open Library "No Cover" Micro-Image
**What goes wrong:** A 1x1 pixel placeholder is saved as the book cover, appearing as a broken/empty image.
**Why it happens:** Open Library returns an image for every request, but if no cover exists, it is a tiny 1x1 placeholder.
**How to avoid:** After `curl -L`, check `wc -c` on the file. If under 1KB, the cover is missing — find an alternative source (Amazon product image, Goodreads, publisher site).
**Warning signs:** Book cover appears as a blank/gray block; downloaded file is ~0KB.

### Pitfall 5: `publishedDate` Format in Frontmatter
**What goes wrong:** Astro's `z.coerce.date()` may behave differently with bare date strings vs ISO datetime strings depending on timezone.
**Why it happens:** `'2019-12-04'` parsed as UTC midnight can appear as Dec 3 in UTC-N timezones.
**How to avoid:** Use `'YYYY-MM-DD'` format consistently (matches all existing posts in the project). Do not add time component.
**Warning signs:** Post date displays one day earlier than expected.
